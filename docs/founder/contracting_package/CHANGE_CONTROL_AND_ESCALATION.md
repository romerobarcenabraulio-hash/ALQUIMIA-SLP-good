# Change Control and Escalation

**Uso:** control de cambios y escalamiento institucional.

## Cuando cambia el alcance

- Nuevo modulo, plataforma o tenant.
- Nuevo documento con efecto institucional.
- Claim de ahorro, captura, reduccion o impacto.
- Uso de datos de tercero/concesionario.
- Solicitud de estudio local.
- Uso de analytics agregada o capa de aprendizaje supervisado publicado.
- Cambio de unidad territorial municipio/ZM.

## Cuando una cifra nueva cambia claims

Si una cifra modifica baseline, CAPEX/OPEX, TIR, rutas, captura, impacto o KPI, se debe registrar fuente, fecha, metodo, confianza, responsable y advertencia. Si contradice evidencia previa, abrir discrepancia.

## Cuando se requiere estudio local

Cuando el claim depende de generacion/composicion RSU, rutas reales, infraestructura fisica, PSP, censo de informales, dictamen juridico o impacto municipal.

## Cuando pausar una recomendacion

Pausar si falta evidencia, hay N insuficiente en capa de aprendizaje supervisado, no hay opt-in, bias check falla, falta founder gate o el cliente quiere tratar sugerencia como decision.

## Escalamiento

| Caso | Escalar a |
|---|---|
| Riesgo comercial/metodologico | Founder |
| Documento juridico, Cabildo, contrato o reforma | Juridico/sindicatura |
| Trazabilidad, auditoria, impugnacion | revision metodologica/contraloria |
| Estandar ESG/GRI/ISO/SDG | revision de estandares/responsable ESG |
| Unidad territorial o schema | validacion de arquitectura/responsable tecnico |
| Promesa previa incorrecta | Founder y responsable de relacion |

## Corregir una promesa previa

1. Registrar promesa en `PROMISE_CONTROL_REGISTER.md`.
2. Clasificar permitido/condicionado/prohibido.
3. Enviar aclaracion escrita si fue externa.
4. Corregir propuesta/SOW.
5. Registrar responsable humano y fecha.

## Registro de decisiones humanas

Toda decision debe guardar: quien decide, rol, fecha, evidencia revisada, decision, limitaciones y si queda condicionada.
