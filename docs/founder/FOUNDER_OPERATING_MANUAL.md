# Founder Operating Manual

**Uso:** manual interno para presentar, vender, defender y operar ALQUIMIA sin exceder sus limites metodologicos.

## 1. Frase institucional

ALQUIMIA es una plataforma institucional para estructurar evidencia municipal de RSU, separar etapas de decision, preparar diagnosticos y borradores trazables, y mostrar brechas antes de que una conclusion sea tratada como oficial.

## 2. Que NO es ALQUIMIA

ALQUIMIA no es:

- certificador oficial;
- auditor externo;
- dictamen juridico;
- aprobacion de Cabildo;
- sustituto del municipio, juridico, consultor, concesionario o founder;
- garantia de ahorro, captura, reduccion o impacto;
- sistema autonomo que cierre gates;
- ML opaco que decide por el cliente.

## 3. Que puede prometerse

Se puede prometer:

- una estructura clara de trabajo por tenant, etapa y gate;
- una primera lectura preliminar con fuente, fecha, metodo y confianza cuando haya fuente disponible;
- distincion visible entre dato validado, inferencia, benchmark, estimacion, estudio local y brecha critica;
- borradores documentales para revision humana;
- trazabilidad metodologica de claims;
- bloqueo de claims cuando falte evidencia minima;
- privacidad por tenant y uso agregado solo con opt-in;
- que capa de aprendizaje supervisado observa y prepara sugerencias solo bajo controles humanos cuando corresponde.

## 4. Que no puede prometerse

No se puede prometer:

- datos oficiales si solo hay inferencia, benchmark o estimacion;
- estudio local si no fue ejecutado y validado;
- certificacion de resultados;
- aprobacion juridica, Cabildo o autoridad;
- ahorro garantizado;
- porcentaje de captura garantizado;
- reduccion asegurada;
- capa de aprendizaje supervisado maduro o predictivo como autoridad;
- cobertura universal de fuentes publicas;
- que flujo de borradores asistidos o capa de aprendizaje supervisado aprueben decisiones;
- que la ZM sustituye al municipio.

## 5. Como explicar las tres plataformas

| Plataforma | Explicacion sobria | Limite de venta |
|---|---|---|
| Validacion `/v` | Reune diagnostico preliminar, fuentes, brechas y evidencia requerida. | No convierte datos preliminares en oficiales. |
| Planeacion `/p` | Ordena plan, cronograma, infraestructura, costos y gobernanza cuando el tenant esta habilitado. | No debe mostrarse como etapa activa sin gate correspondiente. |
| Ejecucion `/e` | Da seguimiento a operacion, KPIs, presupuesto, riesgos y desviaciones. | No sustituye contratos, auditoria externa ni operacion real. |

## 6. Como explicar Plataforma 0

Plataforma 0 es el backoffice interno para administrar tenants, etapas, gates, capabilities, evidencia, documentos y controles de privacidad.

No se presenta como portal publico ni como herramienta para que sistemas internos cierren decisiones. Los gates y cambios de etapa son manuales y requieren evidencia humana.

## 7. Como explicar capa de aprendizaje supervisado

capa de aprendizaje supervisado observa correcciones, outcomes y deltas para detectar patrones trazables. No es autoridad, no firma, no aprueba, no cierra gates y no publica patrones sin opt-in, N suficiente, bias check, founder gate y trazabilidad.

Frase segura:

> capa de aprendizaje supervisado ayuda a observar aprendizaje operativo bajo supervision; no decide por el municipio ni por ALQUIMIA.

## 8. Como explicar flujo de borradores asistidos

flujo de borradores asistidos puede preparar lecturas, recomendaciones o borradores a partir de evidencia disponible. flujo de borradores asistidos no aprueba decisiones politicas, no firma documentos y no convierte un borrador en oficial.

Frase segura:

> flujo de borradores asistidos organiza argumentos y evidencia; la aprobacion sigue siendo humana.

## 9. Como explicar benchmarks

Un benchmark es una referencia comparativa. Sirve para orientar una conversacion, dimensionar una brecha o explicar un rango. No prueba una condicion local.

Frase segura:

> Este benchmark no sustituye el estudio local; solo nos da un punto de comparacion.

## 10. Como explicar falta de estudio local

Si falta estudio local, no se disimula. Se declara brecha critica o brecha recomendada, segun el gate y el claim.

Frase segura:

> Aqui no vamos a inventar una cifra municipal. Falta el estudio local; por eso el claim queda preliminar o bloqueado.

## 11. Como explicar brechas criticas

Una brecha critica es una proteccion metodologica. Evita que una estimacion, inferencia o benchmark se use como verdad municipal.

No se presenta como oportunidad comercial. Se presenta como condicion de evidencia.

## 12. Manejo de objeciones

| Actor | Objecion probable | Respuesta operativa |
|---|---|---|
| Municipio | "Necesito presentarlo rapido." | Se puede presentar como preliminar si lleva advertencias; no como oficial sin evidencia. |
| Concesionario | "La estimacion no coincide." | Se pide evidencia operativa y se actualiza o bloquea el claim segun soporte. |
| Auditor | "Deme trazabilidad." | Se entrega fuente, fecha, metodo, version, responsable y ClaimLedger/provenance. |
| Juridico | "Quien aprobo esto?" | Solo humano autorizado aprueba; sistemas internos preparan, no aprueban. |
| Regidor | "De donde sale esta cifra?" | Se muestra fuente, alcance territorial, metodo y advertencia. |

## 13. Cuando detener una conversacion comercial

Detener o pausar si:

- el prospecto exige presentar estimaciones como oficiales;
- quiere usar benchmark como estudio local;
- pide ocultar brechas criticas;
- pide afirmar certificacion inexistente;
- quiere publicar patrones capa de aprendizaje supervisado sin controles;
- quiere usar datos privados de otro tenant;
- quiere cerrar gate sin evidencia;
- solicita promesas de ahorro/captura/reduccion garantizadas.

La salida correcta es: "Podemos avanzar si lo tratamos como preliminar y dejamos claro que requiere validacion humana/evidencia local."
