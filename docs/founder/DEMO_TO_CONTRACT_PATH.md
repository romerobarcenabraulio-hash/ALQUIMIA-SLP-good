# Demo to Contract Path

**Uso:** ruta interna para avanzar de demo a contrato sin vender claims no sustentados.

## 1. Demo conceptual

| Campo | Regla |
|---|---|
| Que se puede mostrar | Arquitectura por plataformas, Plataforma 0, ejemplo de tenant, advertencias, matriz de evidencia y rutas `/v`, `/p`, `/e`. |
| Que no se puede afirmar | Que el municipio ya tiene diagnostico oficial o resultados garantizados. |
| Evidencia minima | Capturas/demo controlada y explicacion de limites. |
| Decision humana requerida | Founder decide si hay fit para diagnostico preliminar. |
| Documentos necesarios | Manual founder, red lines, audit readiness. |
| Condicion de avance | Prospecto acepta diferencia entre demo y evidencia real. |
| Condicion de bloqueo | Prospecto exige claims oficiales sin datos. |

## 2. Diagnostico preliminar

| Campo | Regla |
|---|---|
| Que se puede mostrar | Datos publicos, inferencias, benchmarks, pendientes, brechas y fuentes disponibles. |
| Que no se puede afirmar | Que las cifras son verdad municipal validada. |
| Evidencia minima | Fuente, fecha, metodo, confianza y advertencia por dato. |
| Decision humana requerida | Founder/responsable tecnico decide que claims quedan preliminares. |
| Documentos necesarios | Claim evidence matrix, glossary, field study status. |
| Condicion de avance | Municipio acepta revisar fuentes y aportar datos. |
| Condicion de bloqueo | Se pide ocultar que una cifra es inferida o benchmark. |

## 3. Diagnostico con datos provistos

| Campo | Regla |
|---|---|
| Que se puede mostrar | Datos municipales, concesionario, contratos, rutas, reglamentos, fuentes y discrepancias. |
| Que no se puede afirmar | Que todo dato provisto es correcto sin revision. |
| Evidencia minima | Archivo, folio, fuente, periodo, unidad, responsable y metodo de captura. |
| Decision humana requerida | Area municipal, concesionario o founder valida/corrige. |
| Documentos necesarios | Technical challenge protocol y readiness checklist. |
| Condicion de avance | Datos tienen responsable y trazabilidad suficiente. |
| Condicion de bloqueo | Datos incompletos se quieren usar como oficiales. |

## 4. Piloto controlado

| Campo | Regla |
|---|---|
| Que se puede mostrar | Tenant en etapa correspondiente, gates, capabilities, documentos bloqueables y seguimiento de brechas. |
| Que no se puede afirmar | Que el piloto equivale a operacion institucional plena. |
| Evidencia minima | Alcance, responsables, gates, evidencia minima, privacidad y opt-in si aplica. |
| Decision humana requerida | Founder y contraparte municipal aceptan alcance. |
| Documentos necesarios | Plataforma 0 spec, readiness checklist, red lines. |
| Condicion de avance | Gates humanos y evidencias acordadas. |
| Condicion de bloqueo | Se quiere avanzar etapa sin evidencia o con gate automatico. |

## 5. Estudio local

| Campo | Regla |
|---|---|
| Que se puede mostrar | Requisitos de cuarteo, rutas, pepenadores, infraestructura, juridico y PSP. |
| Que no se puede afirmar | Que ALQUIMIA ya ejecuto el estudio si no existe evidencia. |
| Evidencia minima | Alcance, fecha, metodo, fuente, responsable, muestra/unidad y limitaciones. |
| Decision humana requerida | Municipio/tercero/founder aceptan o rechazan estudio. |
| Documentos necesarios | Field study status method y claim evidence matrix. |
| Condicion de avance | Estudio validado o brecha aceptada como pendiente. |
| Condicion de bloqueo | Benchmark o inferencia usado como estudio local. |

## 6. Operacion institucional

| Campo | Regla |
|---|---|
| Que se puede mostrar | Gates, seguimiento, documentos revisados, KPIs, riesgos, evidencias y auditoria. |
| Que no se puede afirmar | Que ALQUIMIA sustituye operacion municipal, auditor externo o juridico. |
| Evidencia minima | Contrato, roles, fuentes, estudios, audit logs, revision humana y privacidad. |
| Decision humana requerida | Autoridad municipal/founder/juridico definen uso institucional. |
| Documentos necesarios | Method statement, challenge protocol, readiness checklist, documentos contractuales. |
| Condicion de avance | Responsables y datos operativos activos. |
| Condicion de bloqueo | Falta consentimiento, privacidad, evidencia o autoridad de decision. |

## 7. Contrato recurrente

| Campo | Regla |
|---|---|
| Que se puede mostrar | Alcance de servicio, soporte, actualizacion de datos, revisiones, gates y entregables. |
| Que no se puede afirmar | Certificacion, resultados garantizados, capa de aprendizaje supervisado maduro o automatizacion oficial plena. |
| Evidencia minima | Alcance contractual, SLA si existe, responsabilidades, limites y anexos metodologicos. |
| Decision humana requerida | Founder, cliente y juridico aceptan terminos. |
| Documentos necesarios | Contrato, anexos de metodologia, privacidad, opt-in si aplica, matriz de claims. |
| Condicion de avance | Contrato incorpora limites metodologicos. |
| Condicion de bloqueo | Cliente exige eliminar advertencias o convertir estimaciones en oficiales. |

## Regla de avance

Ninguna etapa avanza por presion comercial. Avanza cuando hay evidencia suficiente, responsable humano identificado y advertencias metodologicas visibles.
